# 高校公寓管理系统 需求分析文档V1
适配：大二学生团队 / 3人协作 / 两周开发周期 / SSM 技术栈课程设计

## 一、项目概述
### 1.1 项目背景与建设目标
本项目为《企业级开发应用》课程设计落地项目，以高校学生公寓日常管理场景为业务载体，严格基于指定技术栈构建单体 Web 应用，覆盖学生入住、日常运维、行政管控全业务流程。
项目以「生产级工程规范」为核心标准，在限定周期内完成可运行、可部署、文档齐全的完整交付，同步落实课程三大培养目标：掌握企业级开发全流程方法论、落地多人敏捷协作、建立工程质量与安全合规意识。

### 1.2 技术栈与选型说明
#### 后端强制技术栈（必选）
- 持久层：MyBatis（动态 SQL、预编译、结果映射）
- 控制层：Spring MVC（请求映射、参数校验、视图解析）
- 切面能力：Spring AOP（操作日志、事务管理、权限校验）
- 项目构建：Maven（依赖管理、多环境配置、打包部署）
- 运行容器：Tomcat 9.x
- 数据库：MySQL 8.0

#### 辅助工具说明
- 允许使用 AI 编码工具辅助生成基础 CRUD 代码、配置模板与语法纠错，但核心业务逻辑、架构设计、安全校验必须由团队自主实现与审核，确保课程能力培养落地。
- 版本控制采用 Git，遵循分支管理与提交规范，支撑多人协作。

### 1.3 文档范围
本文档为项目初版需求基线，明确业务边界、角色权限、功能清单与验收标准，作为后续架构设计、数据库设计、开发排期与测试验收的唯一依据。

## 二、用户角色与权限划分
系统设置三类用户角色，采用「角色-权限」隔离机制，不同角色登录后展示对应功能菜单，越权访问由拦截器统一拦截。

| 角色 | 权限范围 | 核心定位 |
| :--- | :--- | :--- |
| 学生 | 仅可操作与本人相关的功能，查看个人住宿、报修、晚归记录 | 普通业务发起方，系统服务对象 |
| 宿管 | 仅可管理所负责楼栋的日常事务，不可修改系统全局配置 | 一线业务执行方，楼栋级运维 |
| 系统管理员 | 全系统功能权限，可管理所有楼栋、用户、基础配置 | 全局管控方，系统配置与数据统筹 |

## 三、功能需求说明
所有功能按交付优先级分为「必做模块」与「可选扩展模块」；必做模块为两周周期内必须落地的核心内容，保障基础验收通过；可选模块为亮点加分项，视团队进度与能力弹性落地。

### 3.1 必做功能模块（14天周期保障交付）
#### 3.1.1 用户认证与权限管理模块
**对应技术点**：Spring MVC 拦截器 + Spring AOP 权限校验 + 统一返回封装
- 登录/登出：支持三类角色通过账号密码登录，登录状态 Session 管理，安全退出
- 权限拦截：基于角色的菜单动态展示，后端接口级权限校验，未登录/越权访问统一跳转或返回错误
- 个人信息维护：所有角色可查看、修改个人基础信息与登录密码
- 操作日志：所有管理员、宿管的增删改操作，通过 Spring AOP 自动记录操作人、操作时间、操作内容，支持追溯

#### 3.1.2 宿舍基础信息管理模块
**对应技术点**：MyBatis 基础 CRUD + 动态查询 + 事务管理
- 楼栋管理：管理员可新增、编辑、删除楼栋信息（楼栋号、楼栋名称、楼层数、所属区域、负责宿管）
- 房间管理：管理员可按楼栋维护房间信息（房间号、所属楼栋、楼层、床位数、房间类型），支持按楼栋、房间号筛选查询
- 床位管理：管理员维护床位信息（床位号、所属房间、入住状态），支持批量初始化房间床位
- 数据关联：楼栋-房间-床位三级关联，删除时做存在性校验，避免脏数据

#### 3.1.3 入住与调宿管理模块
**对应技术点**：多表关联查询 + 事务控制 + 参数校验
- 入住登记：管理员/宿管可为学生办理入住，分配空闲床位，自动更新床位状态，生成入住记录
- 入住信息查询：学生可查看本人当前住宿信息；宿管可查看本楼栋所有入住学生列表；管理员可全局查询
- 调宿申请：学生提交调宿申请，填写目标房间与申请原因
- 调宿审批：管理员/宿管审核调宿申请，审批通过后自动调整床位占用状态、生成调宿记录，审批驳回需填写原因
- 退宿办理：管理员/宿管为学生办理退宿，释放床位，更新入住记录状态

#### 3.1.4 报修管理模块
**对应技术点**：状态流转 + 分页查询 + 前后端联调
- 报修提交：学生填写报修单（报修类型、报修内容、所在房间、联系电话），提交后状态为「待处理」
- 报修处理：宿管查看本楼栋报修单，可标记为「处理中」「已完成」，填写处理结果；管理员可查看全部报修单
- 报修查询：学生可查看本人报修记录与处理进度；支持按状态、时间、楼栋筛选
- 超时提醒：列表中对超过 24 小时未处理的报修单做高亮标识

#### 3.1.5 晚归登记管理模块
**对应技术点**：条件查询 + 数据统计 + AOP 日志记录
- 晚归录入：宿管登记晚归学生信息（学号、姓名、晚归时间、晚归原因、所属楼栋），自动关联学生住宿信息
- 晚归记录查询：学生查看本人晚归记录；宿管查看本楼栋晚归记录，支持按日期、学号筛选；管理员全局查询
- 基础统计：按楼栋统计月度晚归人次，简单列表展示

#### 3.1.6 访客登记管理模块
**对应技术点**：单表 CRUD + 时间范围查询
- 访客录入：宿管登记访客信息（访客姓名、身份证号、被访学生学号、来访时间、离开时间、来访事由）
- 访客记录查询：宿管查看本楼栋访客记录，支持按来访日期、被访人筛选；管理员全局查询
- 离校确认：访客离开时宿管标记离开时间，更新记录状态

### 3.2 可选扩展功能模块（周期充裕时落地）
#### 3.2.1 卫生检查模块
- 宿管按房间录入卫生检查评分、检查日期、备注，支持按楼栋、周/月统计平均分
- 学生可查看本人所在房间的历史检查记录

#### 3.2.2 公告通知模块
- 管理员发布公寓公告（标题、内容、发布时间、生效范围），支持置顶、下架
- 学生、宿管登录后可查看公告列表与详情

#### 3.2.3 数据统计看板
- 管理员首页展示核心数据：入住率、本月报修量、本月晚归人次、访客量
- 简单柱状图/折线图展示月度业务趋势，结合 ECharts 实现

#### 3.2.4 系统增强特性
- 登录图形验证码、密码加密存储（MD5/SHA256）
- 数据批量导入导出（Excel）
- 表单前端+后端双重参数校验
- 敏感操作二次确认

### 3.3 前端技术方案（二选一）
| 方案 | 技术选型 | 适用场景 | 学习成本 |
| :--- | :--- | :--- | :--- |
| 方案A（必做推荐） | JSP + Bootstrap + JSTL | 贴合课程教学范围，与 Spring MVC 原生适配，无需前后端分离部署 | 低 |
| 方案B（可选进阶） | Vue 2.x + Element UI +  Axios | 前后端分离架构，后端输出 RESTful 接口，更贴近当前企业主流技术栈 | 中 |

> 建议：首周优先采用 JSP 方案跑通核心流程，若进度提前可切换为 Vue 前端方案，提升项目亮点。

## 四、核心业务流程
### 4.1 学生入住办理流程
1. 管理员提前维护楼栋、房间、床位基础数据，确保空闲床位可用
2. 管理员/宿管录入学生学号、姓名等基础信息，选择空闲床位
3. 系统校验床位状态，执行事务：生成入住记录 + 更新床位状态为「已入住」
4. 办理完成，学生账号可查看个人住宿信息

### 4.2 报修处理全流程
1. 学生登录系统，填写报修单并提交，状态为「待处理」
2. 对应楼栋宿管登录系统，查看本楼栋待处理报修单
3. 宿管上门处理后，更新报修状态为「已完成」，填写处理结果
4. 学生可查看报修处理进度与结果

### 4.3 调宿申请审批流程
1. 学生提交调宿申请，选择目标空闲床位、填写申请原因
2. 管理员/宿管在待审批列表中查看申请
3. 审批通过：系统事务性释放原床位、占用新床位、更新申请状态、生成调宿记录
4. 审批驳回：填写驳回原因，申请结束，原住宿信息不变

### 4.4 访客登记流程
1. 访客到访，宿管核实被访学生信息
2. 宿管录入访客身份信息与来访事由，登记来访时间
3. 访客离开时，宿管登记离开时间，闭环访客记录

## 五、非功能需求
### 5.1 性能需求
- 单接口响应时间 ≤ 1s，列表查询支持分页，默认每页 10 条
- 数据库核心查询字段建立索引，避免全表扫描
- 支持 50 人同时在线使用，无明显卡顿与数据错误

### 5.2 安全需求
- 所有 SQL 通过 MyBatis 预编译实现，杜绝 SQL 注入风险
- 后端接口统一做权限校验，禁止越权访问他人数据与其他角色功能
- 密码加密存储，禁止明文保存用户密码
- 输入参数做合法性校验，过滤特殊字符，避免 XSS 风险

### 5.3 可用性与可维护性需求
- 采用 MVC 三层架构，分层清晰，职责明确，耦合度低
- 统一返回格式、统一异常处理、统一错误码，便于排查问题
- 核心业务方法添加事务注解，保证数据一致性
- 代码遵循阿里巴巴 Java 开发规范，命名规范、注释完整

### 5.4 可追溯需求
- 所有管理端增删改操作通过 Spring AOP 自动记录操作日志
- 业务数据保留操作人与操作时间字段，支持审计追溯
- 关键状态变更保留历史记录，不可物理删除核心业务数据

## 六、需求边界与交付说明
### 6.1 本期交付范围
- 必做 6 大业务模块完整功能，三类角色权限隔离
- 生产级基础特性：统一返回、全局异常、AOP 日志、事务管理、权限拦截
- 配套数据库脚本、可部署 WAR 包、全链路项目文档

### 6.2 暂不实现的功能
- 人脸识别、门禁系统对接等硬件集成功能
- 在线支付、水电费缴纳等财务相关功能
- 分布式架构、微服务、消息队列等超出课程技术栈的架构
- 复杂大数据分析、多维度智能报表

### 6.3 验收参考标准
1. 功能完整性：必做模块 100% 落地，核心业务流程可完整跑通
2. 工程规范性：SSM 架构分层合理，Spring AOP 正确应用，MyBatis 写法规范
3. 质量合规性：无阻断性 Bug，无越权、SQL 注入等安全漏洞，核心数据一致性
4. 文档完整性：需求、设计、测试、部署、用户手册齐全，符合企业文档规范

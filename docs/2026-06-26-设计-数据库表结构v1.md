# 高校公寓管理系统 数据库表结构设计（全量必做业务覆盖版）
适配：SSM 技术栈 / MySQL 8.0 / 大二课程设计 / 生产级工程规范

## 一、设计原则与整体说明
### 1.1 设计依据
完全覆盖需求文档中**6大必做业务模块**+三类用户角色体系+AOP操作日志需求，遵循第三范式设计，兼顾业务完整性、查询性能与开发实现成本，适配两周开发周期与学生团队技术水平。

### 1.2 命名与规范约定
- 表名统一采用 `模块前缀_业务名称` 格式，系统表前缀 `sys_`，公寓业务表前缀 `dorm_`，全部小写+下划线分隔
- 主键统一命名为 `xxx_id`，采用 `BIGINT` 自增类型；状态、类型字段用 `TINYINT` 存储，配套枚举字典
- 所有业务表默认包含创建时间、更新时间字段；基础配置表支持逻辑删除，核心业务表保留完整历史记录
- **逻辑外键设计**：表间通过ID字段建立逻辑关联，不强制创建物理外键，降低开发调试复杂度，通过业务代码+事务保证数据参照完整性
- 核心查询字段建立普通索引，保障分页、条件查询性能

| 表分类 | 表名 | 对应业务模块 | 核心作用 |

| `username` | VARCHAR(50) | NOT NULL, UNIQUE | 登录账号 | 学生用学号、宿管/管理员用工号 |
| `password` | VARCHAR(100) | NOT NULL | 登录密码 | MD5/SHA256加密存储，禁止明文 |
| `real_name` | VARCHAR(20) | NOT NULL | 真实姓名 | |
| `role_type` | TINYINT | NOT NULL, DEFAULT 3 | 角色类型 | 1=系统管理员 2=宿管 3=学生 |
| `gender` | TINYINT | NULL | 性别 | 1=男 2=女 |
| `phone` | VARCHAR(20) | NULL | 联系电话 | |
| `grade` | VARCHAR(20) | NULL | 年级 | 学生专属字段 |
| `major` | VARCHAR(50) | NULL | 专业 | 学生专属字段 |
| `class_name` | VARCHAR(50) | NULL | 班级 | 学生专属字段 |
| `status` | TINYINT | NOT NULL, DEFAULT 1 | 账号状态 | 1=正常 0=禁用 |
| `create_time` | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 | |
| `update_time` | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 | |
| `is_deleted` | TINYINT | NOT NULL, DEFAULT 0 | 逻辑删除标记 | 0=未删除 1=已删除 |

## 六、补充设计说明
### 6.1 核心事务联动规则
表间状态变更需在事务中原子执行，保障数据一致性：
1. **入住办理**：新增入住记录 + 更新床位状态为「已入住」，原子执行
2. **退宿办理**：更新入住记录状态为「已退宿」+ 填写退宿时间 + 更新床位状态为「空闲」，原子执行
3. **调宿审批通过**：更新原入住记录（退宿标记）+ 新建新入住记录 + 原床位置空闲 + 目标床位置已入住，原子执行
4. **删除楼栋/房间**：校验是否存在在住学生，存在则禁止删除，避免脏数据

### 6.2 数据字典枚举汇总
| 字段 | 枚举值说明 |
| :--- | :--- |
| `role_type` | 1=管理员 2=宿管 3=学生 |
| `bed_status` | 0=空闲 1=已入住 2=维修禁用 |
| `checkin_status` | 1=在住 2=已退宿 |
| `audit_status` | 0=待审批 1=已通过 2=已驳回 |
| `repair_type` | 1=水电故障 2=家具损坏 3=门窗故障 4=其他 |
| `repair_status` | 0=待处理 1=处理中 2=已完成 |

2. 预置2-3栋测试楼栋、每栋5-10个房间、每房间4个床位，便于联调测试
3. 预置3-5个测试学生账号、2个宿管账号，覆盖三类角色测试场景

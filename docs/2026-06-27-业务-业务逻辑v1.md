## 一、用户认证与权限模块（sys_user）

### 1. 学生账号创建
- **创建账号时必须录入完整信息**：年级、专业、班级等学生字段均需提供。
- **初始密码**：采用「年级 + 手机号」拼接后 MD5 加密存储，无需管理员手动设置。
- **默认密码生成规则**：`UserServiceImpl.generateDefaultPassword(grade, phone)` 方法将年级和手机号直接拼接（如"202413800138000"），然后通过 `Md5Util.encrypt()` 加密后存储到 `password` 字段。

### 2. 首次登录强制修改密码
- **判定方式**：后端比对登录密码的 MD5 是否等于默认密码（年级+手机号的 MD5 值），若相等则强制跳转修改密码页面。
- **具体实现**：`UserServiceImpl.needChangePassword(SysUser user)` 方法执行以下步骤：
  1. 查询完整用户信息（包含密码字段）
  2. 调用 `generateDefaultPassword(grade, phone)` 生成默认密码明文
  3. 调用 `Md5Util.verify(defaultPassword, fullUser.getPassword())` 比对
  4. 返回 `true` 表示需要修改密码
- 不增加 `need_change_password` 字段，避免冗余。

### 3. 密码修改
- **修改密码不需要验证旧密码**，简化流程。
- 学生首次登录后，可自行设置新密码。
- **具体实现**：`UserServiceImpl.updatePassword(PasswordDTO dto, Long userId)` 方法执行以下步骤：
  1. 校验新密码与确认密码一致
  2. 调用 `Md5Util.encrypt(dto.getNewPassword())` 加密
  3. 调用 `sysUserMapper.updatePassword(userId, encryptedPassword)` 更新

### 4. 账户锁定
- **不启用登录失败锁定机制**，无需错误次数限制和自动解锁。

### 5. 个人信息修改权限
- **学生**：仅允许修改密码和联系电话。
- **管理员**：拥有全字段编辑权限，可对学生信息进行纠错与调整。

### 6. 用户状态管理
- **状态类型**：正常（status=1）、禁用（status=0）。
- **状态切换**：管理员可通过 `AdminUserController.toggleUserStatus(userId)` 接口切换用户状态。
- **具体实现**：`UserServiceImpl.toggleUserStatus(Long userId)` 方法执行以下步骤：
  1. 查询用户当前状态
  2. 若当前为正常，则切换为禁用；若当前为禁用，则切换为正常
  3. 调用 `sysUserMapper.updateStatus(userId, newStatus)` 更新
- **禁用账号登录拦截**：`UserServiceImpl.login()` 方法校验账号状态，若为禁用则抛出异常，提示"账号已被禁用，请联系管理员"。

### 7. 操作日志记录
- **登录操作**：`AuthController.login()` 方法标注 `@OperLog(module = "用户认证", type = OperTypeEnum.LOGIN, desc = "用户登录")`。
- **登出操作**：`AuthController.logout()` 方法标注 `@OperLog(module = "用户认证", type = OperTypeEnum.LOGOUT, desc = "用户登出")`。
- **失败操作记录**：所有标注 `@OperLog` 的方法，无论成功或失败，均记录到 `sys_oper_log` 表。失败时在操作描述中追加异常信息（截取前500字符）。
- **日志记录不影响主业务**：日志记录失败时，仅打印错误日志，不阻断业务流程。

### 8. 角色与权限总览
- 管理员：可管理所有数据，全字段编辑用户信息。
- 宿管：仅可查看本人负责楼栋的相关信息（房间、床位、入住等），无新增/编辑/删除权限（除后述业务操作如入住办理、报修、晚归、访客等）。
- 学生：查看/修改个人密码和电话，申请调宿、报修，查看申请进度。

## 二、宿舍楼栋与床位管理

### 1. 宿管权限
- 宿管仅可**查看本人负责楼栋**的房间、床位列表（用于入住办理、报修核对等），前端不提供新增、编辑、删除入口。
- **自动获取负责楼栋**：宿管端查询接口（如 `DormCheckinController.getCheckinPage()`、`DormRepairController.getRepairPage()` 等）自动通过 `BuildingService.getBuildingsByManagerId(managerId)` 查询宿管负责的楼栋列表，取第一个楼栋作为默认查询条件。
- **无楼栋时返回明确提示**：若宿管未负责任何楼栋，接口返回 `Result.fail("您暂未负责任何楼栋，请联系管理员")`。

### 2. 批量初始化床位
- **保留批量初始化功能**，支持为指定房间批量生成床位。
- **床位号生成规则**：默认按数字序号（1号床、2号床…N号床），可扩展为上下铺命名规则，但当前版本用数字序号即可。
- **具体实现**：`BedServiceImpl.batchAddBeds(BatchBedDTO dto, Long operatorId)` 方法执行以下步骤：
  1. 校验房间存在（`dormRoomMapper.selectById(dto.getRoomId())`）
  2. 校验床位数大于0
  3. 循环生成 DormBed 对象，bedNo 格式为 `i + "号床"`（i 从1开始）
  4. 调用 `dormBedMapper.batchInsert(beds)` 批量插入
- **事务一致性**：批量新增时需校验房间是否存在、床位数合法，任一失败则整体回滚，保证原子性。

### 3. 数据删除策略
- **存在关联数据（下级房间/床位）时，禁止删除**，给出明确提示。
- **无关联数据时**，执行**逻辑删除**（标记 `is_deleted` 等字段，而不是物理删除）。
- **双重保障**：前端弹窗二次确认；后端 Service 层在事务内做最终校验并执行删除，保证校验与操作的原子性。

## 三、入住与调宿

### 1. 入住登记
- **操作流程**：管理员/宿管先选择学生，再选择空闲床位，确认后提交。
- **前置校验**：
  - 学生当前无有效在住记录；
  - 学生账号状态正常（未删除、未被禁用）。
- **床位选择**：下拉列表仅展示状态为「空闲」的床位，并支持按楼栋、房间筛选。
- **事务提交**：原子操作，同时生成入住记录并将床位状态更新为「已入住」。

### 2. 调宿申请（学生端）
- **申请内容**：学生选择目标床位 + 填写申请原因。
- **自动带入**：系统自动填充当前原床位信息（楼栋、房间、床位），学生无需手动输入。
- **前置校验**：
  - 学生当前必须有有效在住记录；
  - 同一学生仅允许存在一条「待审批」状态的调宿申请；
  - 目标床位必须为空闲状态。
- **申请原因**：必填，有长度限制。
- **学生查看**：可查看自己申请的审批进度和结果（状态、审批意见等）。

### 3. 调宿审批与自动执行（管理员/宿管端）
- **审批通过后自动切换**：
  - 释放原床位（状态置为空闲）；
  - 占用新床位（状态置为已入住）；
  - 创建新入住记录（而非更新原记录），保留历史入住记录。
- **具体实现**：`MoveApplyServiceImpl.auditApproved()` 方法在事务内执行以下步骤：
  1. 校验目标床位仍为空闲
  2. 获取原入住记录
  3. 释放原床位（`bedMapper.updateStatus(originalBedId, FREE)`）
  4. 占用目标床位（`bedMapper.updateStatus(targetBedId, OCCUPIED)`）
  5. 更新原入住记录为已退宿（`checkinStatus = CHECKED_OUT`）
  6. 创建新入住记录（`checkinStatus = LIVING`）
  7. 更新申请状态为已通过
- **列表展示**：申请时间、原床位信息、目标床位信息、申请原因、审批状态、审批意见。
- **状态筛选**：支持按审批状态（待审批/已通过/已驳回）筛选。
- **撤销限制**：仅「待审批」状态可撤销，已通过或已驳回的申请不可操作。
- **撤销后状态**：学生撤销调宿申请后，申请状态变为「已驳回」，审批意见自动填入"申请人主动撤销"。

## 四、报修管理、晚归登记与访客登记

### 1. 报修管理
#### 学生提交报修
- **房间信息自动带入**：根据学生当前在住记录自动填入楼栋号和房间号，字段设为只读不可修改。
- **后端防御**：提交时再次查询学生在住状态及房间归属，完全不依赖前端传入的房间 ID，避免篡改。
- **具体实现**：`RepairServiceImpl.submitRepair(RepairDTO dto, Long studentId)` 方法执行以下步骤：
  1. 查询学生在住记录（`checkinRecordMapper.selectActiveByStudentId(studentId)`）
  2. 通过床位查询房间ID（`bedMapper.selectById(checkinRecord.getBedId())`）
  3. 使用 `bed.getRoomId()` 设置报修记录的 roomId

#### 宿管处理报修
- 操作分两步：
  1. **接单处理**：将报修状态改为「处理中」，回填处理人 ID（当前宿管）。
  2. **完结报修**：将状态改为「已完成」，此时**必须填写处理结果**（多行文本框，可记录维修内容、更换配件等），系统自动回填完成时间。
- 只有在「已完成」状态下才可填写处理结果。

#### 报修超时提醒
- **超时阈值**：24小时。
- **判定逻辑**：`RepairServiceImpl.convertToVO()` 方法判断报修状态为「待处理」且当前时间超过 `submitTime + 24小时`，则设置 `isTimeout = true`。
- **前端展示**：超时的报修记录在列表中高亮标识。

### 2. 晚归登记
#### 宿管录入
- **录入表单**：仅需提供「学生学号/姓名选择」「晚归时间」「晚归原因」三项。
- **自动关联**：后端通过学生最新有效在住记录自动获取 `building_id`，存入晚归表，不接收前端传入的楼栋参数。
- **具体实现**：`LateReturnServiceImpl.addRecord(LateReturnDTO dto, Long registrarId)` 方法执行以下步骤：
  1. 查询学生在住记录（`checkinRecordMapper.selectActiveByStudentId(dto.getStudentId())`）
  2. 通过床位查询房间（`bedMapper.selectById(checkinRecord.getBedId())`）
  3. 通过房间查询楼栋（`roomMapper.selectById(bed.getRoomId())`）
  4. 使用 `room.getBuildingId()` 设置晚归记录的 buildingId
- 学生列表供选择时，可搜索学号或姓名。

#### 晚归统计
- **维度**：按月统计，按楼栋分组，输出「楼栋名称」和「晚归总人次」两列。
- **权限隔离**：
  - 管理员：可查看所有楼栋统计。
  - 宿管：仅可查看自己负责楼栋的统计数据。

### 3. 访客登记
- **来访时**：记录访客信息、被访学生、访问时间等（具体录入字段未详细说明，但聚焦离开操作）。
- **自动关联楼栋**：后端通过被访学生在住记录自动获取 `building_id`，不接收前端传入的楼栋参数。
- **具体实现**：`VisitorServiceImpl.addVisitor(VisitorDTO dto, Long registrarId)` 方法执行以下步骤：
  1. 查询被访学生在住记录（`checkinRecordMapper.selectActiveByStudentId(dto.getStudentId())`）
  2. 通过床位查询房间，再查询楼栋
  3. 使用 `room.getBuildingId()` 设置访客记录的 buildingId
- **离开操作**：宿管在页面上看到"未离开访客列表"，每条记录后有「确认离开」按钮，点击后系统**自动填充当前系统时间**为离开时间。
- **具体实现**：`VisitorServiceImpl.confirmLeave(Long visitorId)` 方法执行以下步骤：
  1. 校验访客记录存在且未离开（`leaveTime == null`）
  2. 调用 `visitorMapper.updateLeaveTime(visitorId, LocalDateTime.now())` 更新离开时间
- **时间修正**：仅管理员有权限编辑历史离开时间；宿管只能进行一键确认离开操作，不能修改时间。
- **身份证号脱敏**：访客身份证号在返回给前端时自动脱敏处理。
- **具体实现**：`VisitorServiceImpl.convertToVO()` 方法对18位身份证号进行脱敏，保留前6位和后4位，中间用 `********` 替代（如"110101********1234"）。

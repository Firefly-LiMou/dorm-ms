# **必须严格遵循以下要求**

## Git 提交规范
### 格式
每次完成一阶段修改后必须立即提交，提交信息格式：
type: description
type 取值必须为以下之一：

- feat: 新功能
- fix: 修复缺陷
- refactor: 重构（不改变外部行为）
- style: 样式调整
- docs: 文档变更
- chore: 构建、工具、依赖变更
- test: 测试相关

description 使用中文。多行提交信息第二行起每行以 “- ” 开头列出具体变更点。

示例：
feat: 新增用户登录页
fix: 修复令牌过期导致的重定向循环问题

- 在 401 响应中清除令牌
- 重定向至登录页面，使用原始的 URL 作为重定向参数
- 添加拦截器测试

### 提交说明

当用户拒绝提交命令时，说明可能：

- 未进行测试
- 用户测试不通过
- 提交不规范

此时应该立即停止提交，并**结束对话**，等待用户说明。


## 复杂需求分析原则
对于用户提出的复杂要求，在动手写代码之前，必须先完成以下四项分析。
如果任意一项无法确定，必须主动向用户逐一提问，直到全部明确为止。

1. **目标（Goal）**：到底要完成什么？用户期望的最终结果是什么？
2. **约束（Constraints）**：必须遵守什么？技术栈限制、业务规则、性能要求、兼容性要求？
3. **验证（Verification）**：怎么判断它真的做完了？具体的验收标准、测试用例、可观测的行为？
4. **输出（Output）**：最后要交付什么？哪些文件、什么格式？

### 分析要求
- 每接到一个新任务，先在回复中列出你对这四项的理解。
- 如果用户的要求模糊，不要猜测，直接提问。例如：
  - “你希望在哪个页面展示这个功能？”
  - “对性能有特殊要求吗？比如首屏加载时间？”
  - “验收标准是什么？是否有具体的测试用例？”
- 直到四项全部明确后，才开始编码。


## 任务执行流程
不要一上来就挂大目标、直接写大段代码。必须遵循以下流程：

1. **拆解任务**：将用户需求拆成可独立验证的小步骤（每个步骤 <= 一个文件的改动）
2. **逐步骤执行**：每完成一个步骤，立即检查：
   - 是否通过所有测试
   - 是否符合约束
   - 是否引入新错误
3. **验证后提交**：确认该步骤无误后，按 Git 提交规范提交，然后进入下一步。
4. **进度同步**：每完成一个步骤后，简要告知用户进度（如“已完成步骤 1/4：创建 API 路由”）。


## 架构变更决策（重要）
触发条件：需求涉及**模块边界、数据流向、关键抽象（接口/基类）、部署边界**的新增或修改等**架构级别**的改动。

必须执行以下流程：
1. 暂停“最小改动”思维。
2. 提出至少两个方案，对照以下三维度评估：
- **未来目标的支持性与适配性**
    - 前瞻性：当前这个改动，是否为未来 规划或可预见的需求提供了平滑扩展点。
    - 可演进性：是否允许在未来以增量方式替换、升级或扩展该部分架构，而不触发大规模重构。
- **项目规范与代码规范**
   - 一致性：是否与项目现有架构约定、命名、分层职责一致。
   - 约束合规：必须校验改动是否满足现有的技术约束。任何架构改动都不能破坏这些已确立的边界。
   - 可测试性：新架构必须保证每个层级、每个关键逻辑可以独立测试。
- **职责边界**
   - 单一职责原则：每一层（或每个模块、每个组件）只处理本层逻辑。
   - 依赖方向正确：依赖方向从高层指向低层，禁止反向依赖。
   - 接口隔离：跨层数据传递必须使用 DTO 或适配接口，禁止直接暴露内部结构。
3. 记录最终方案选择理由。

## 文档书写规范

### 通用命名规范

日期+范围+具体类型+版本号

如“2026-06-20-前端架构设计-页面风格设计v1.md“

### 架构设计文档
- 尽量不包含完整代码实现。可包含：
  - 导出函数、类、接口的声明（签名）
  - 关键数据结构定义
  - 模块间调用关系图
- 必须包含以下章节：
  1. 设计目标（量化指标）
  2. 约束条件
  3. 模块职责与边界
  4. 接口契约（输入、输出、错误码）
  5. 数据流向
  6. 扩展点与未来演进计划
- 术语表：定义文档中所有领域术语，全文统一，不得添加无用修饰。

### 通用写作要求
- 每句话描述一个行为，禁止用逗号或“和”、“以及”连接多个独立需求。
- 强制要求使用“必须”、“完全”；建议性内容使用“宜”、“可”并标注优先级。
- 禁止模糊修饰词（非常、很、尽量、可能、或许、大概、基本、一些、等、等等、快速、稳定、友好）。性能等指标必须量化。
- 形容词必须附带检验标准。例如：“正确”需说明判定方法。
- 时间约束精确到数字（“1 秒内”、“50ms 内”）。
- 接口交互须指明具体接口名称或路径，不得使用“相关接口”等描述。


## 标记规范
### 标记约定
代码中需要标注待办、缺陷等信息时，使用以下大写标记，后跟说明：
- TODO: 计划后续完成的功能
- BUG: 已知缺陷，需修复
- TAG: 版本/迭代标记
- DONE: 已完成但需要保留记录（一般无需保留记录）
- MARK: 强调重点逻辑
- TEST: 测试代码或测试点
- XXX: 临时方案，必须附上风险说明

### 书写要求
- 标记必须完全大写，后跟英文冒号及一个空格。
- 每个标记独占一行，置于相关代码块上方。
- 标记后必须有说明，不得留空。
- 禁止使用“可能”、“待优化”等模糊表述，必须明确具体动作或缺陷现象。

# 高校公寓管理系统 测试设计方案V1

## 一、设计目标

### 1.1 量化指标
- 核心业务Service方法测试覆盖率达到80%以上
- 每个测试用例执行时间控制在500ms以内
- 测试数据隔离，每个测试用例执行后自动回滚，不影响其他用例

### 1.2 约束条件
- 测试框架：JUnit 5 + Mockito 5.x
- Spring集成测试：spring-test 5.3.x
- 数据库：MySQL 8.0（复用现有建表脚本和测试数据脚本）
- 测试命名规范：`方法名_场景_预期结果`，如 `checkin_学生已有在住_抛出异常`

## 四、测试策略

### 4.1 单元测试策略（Service层）

**测试方式**：使用Mockito mock掉DAO层依赖，仅测试Service层业务逻辑。

**测试基类设计**：
```java
@ExtendWith(MockitoExtension.class)
public abstract class BaseServiceTest {
    // 通用测试辅助方法
}
```

**测试覆盖维度**：
- 正常路径（happy path）：参数合法、状态满足，验证业务结果
- 异常路径：参数非法、状态不满足，验证BusinessException抛出及错误码
- 边界条件：null参数、空集合、状态临界值

### 4.2 测试数据管理

**原则**：每个测试用例自行构造测试数据，不依赖外部数据文件。

**实体构造方式**：使用new + setter构造，不引入Builder模式（符合项目Lombok规范）。

## 五、核心测试用例设计

### 5.1 CheckinServiceImpl

#### checkin() - 办理入住

| 用例 | 场景 | 预期结果 |
|------|------|----------|
| checkin_正常入住 | 学生存在且状态正常、无在住记录、床位空闲 | 插入入住记录、更新床位状态为已入住 |
| checkin_学生不存在 | studentId对应用户不存在 | 抛出BusinessException，code=404 |
| checkin_学生状态异常 | 学生状态为DISABLED | 抛出BusinessException，code=400 |
| checkin_已有在住记录 | 学生当前有在住记录 | 抛出BusinessException，code=409 |
| checkin_床位不存在 | bedId对应床位不存在 | 抛出BusinessException，code=404 |
| checkin_床位已被占用 | 床位状态为OCCUPIED | 抛出BusinessException，code=409 |

#### checkout() - 办理退宿

| 用例 | 场景 | 预期结果 |
|------|------|----------|
| checkout_正常退宿 | 记录存在且状态为LIVING | 更新记录状态为已退宿、释放床位 |
| checkout_记录不存在 | checkinId对应记录不存在 | 抛出BusinessException，code=404 |
| checkout_状态异常 | 记录状态为CHECKED_OUT | 抛出BusinessException，code=400 |

### 5.2 MoveApplyServiceImpl

#### apply() - 提交调宿申请

| 用例 | 场景 | 预期结果 |
|------|------|----------|
| apply_正常提交 | 有在住记录、无待审批申请、目标床位空闲 | 插入调宿申请记录 |
| apply_无在住记录 | 学生当前无在住记录 | 抛出BusinessException，code=400 |
| apply_已有待审批申请 | 学生已有pending状态的申请 | 抛出BusinessException，code=409 |
| apply_目标床位不存在 | targetBedId对应床位不存在 | 抛出BusinessException，code=404 |
| apply_目标床位被占用 | 目标床位状态为OCCUPIED | 抛出BusinessException，code=409 |

#### audit() - 审批调宿申请

| 用例 | 场景 | 预期结果 |
|------|------|----------|
| audit_审批通过 | 申请存在且状态为PENDING、目标床位空闲 | 释放原床位、占用目标床位、原记录退宿、创建新入住记录、更新申请状态 |
| audit_审批驳回 | 申请存在且状态为PENDING | 仅更新申请状态为REJECTED |
| audit_申请不存在 | applyId对应申请不存在 | 抛出BusinessException，code=404 |
| audit_状态非待审批 | 申请状态为APPROVED或REJECTED | 抛出BusinessException，code=400 |
| audit_通过时目标床位被占用 | 审批通过但目标床位已被他人占用 | 抛出BusinessException，code=409（事务回滚） |

### 5.3 UserServiceImpl

#### login() - 用户登录

| 用例 | 场景 | 预期结果 |
|------|------|----------|
| login_正常登录 | 用户名存在、密码正确、状态正常 | 返回用户信息（密码字段为null） |
| login_用户不存在 | 用户名不存在 | 抛出BusinessException，code=401 |
| login_密码错误 | 用户名存在但密码不匹配 | 抛出BusinessException，code=401 |
| login_账号禁用 | 用户状态为DISABLED | 抛出BusinessException，code=401 |

#### addUser() - 新增用户

| 用例 | 场景 | 预期结果 |
|------|------|----------|
| addUser_正常新增 | 用户名非空且不重复 | 插入用户记录，密码为huuc+手机号的MD5 |
| addUser_用户名为空 | username为null或空字符串 | 抛出BusinessException，code=400 |
| addUser_用户名重复 | 数据库已存在同名用户 | 抛出BusinessException，code=409 |

### 5.4 RepairServiceImpl

#### submitRepair() - 提交报修

| 用例 | 场景 | 预期结果 |
|------|------|----------|
| submitRepair_正常提交 | 学生有在住记录 | 插入报修记录，状态为PENDING，自动关联房间 |
| submitRepair_无在住记录 | 学生当前无在住记录 | 抛出BusinessException，code=400 |

#### handleRepair() - 接单处理

| 用例 | 场景 | 预期结果 |
|------|------|----------|
| handleRepair_正常接单 | 记录存在且状态为PENDING | 更新状态为PROCESSING，回填处理人 |
| handleRepair_记录不存在 | repairId对应记录不存在 | 抛出BusinessException，code=404 |
| handleRepair_状态非待处理 | 状态为PROCESSING或COMPLETED | 抛出BusinessException，code=400 |

#### completeRepair() - 完结报修

| 用例 | 场景 | 预期结果 |
|------|------|----------|
| completeRepair_正常完结 | 记录存在且状态为PROCESSING | 更新状态为COMPLETED，回填处理结果和完成时间 |
| completeRepair_状态非处理中 | 状态为PENDING或COMPLETED | 抛出BusinessException，code=400 |

### 5.5 Md5Util

| 用例 | 场景 | 预期结果 |
|------|------|----------|
| encrypt_正常加密 | 输入非空字符串 | 返回32位小写十六进制MD5值 |
| encrypt_输入null | 输入为null | 返回null |
| verify_密码匹配 | 明文密码与加密后密码匹配 | 返回true |
| verify_密码不匹配 | 明文密码与加密后密码不匹配 | 返回false |
| verify_输入null | password或encryptedPassword为null | 返回false |

## 八、验收标准

### 8.1 功能验收
- 所有测试用例通过，无失败或跳过
- 测试数据隔离，多次执行结果一致

### 8.2 规范验收
- 测试命名符合`方法名_场景_预期结果`规范
- 每个测试用例仅验证一个行为
- 测试代码无硬编码数据库依赖（DAO层通过mock隔离）

### 8.3 覆盖验收
- 阶段二完成后，核心业务方法测试覆盖率达到80%以上
- 覆盖率计算范围：CheckinServiceImpl、MoveApplyServiceImpl、UserServiceImpl、RepairServiceImpl中的写方法（非纯查询）

# 高校公寓管理系统 后端架构设计文档
适配：SSM 单体架构 / 大二课程设计 / 3人团队协作 / 两周开发周期

## 一、架构设计概述
### 1.1 设计背景与目标
本文档为高校公寓管理系统的后端技术架构总纲，严格遵循《企业级开发应用》课程设计要求，基于指定技术栈 `MyBatis + Spring MVC + Spring AOP + Maven + Tomcat` 构建单体企业级 Web 应用。
架构设计以「**分层解耦、规范落地、可控复杂度、适配学生能力**」为核心导向，既满足生产级工程的质量标准（权限隔离、事务一致性、操作可追溯、异常统一处理），又兼顾两周开发周期与大二学生技术基础，避免过度设计，保障项目可按期交付。

### 1.2 核心设计原则
1. **单一职责与分层解耦**：严格遵循 MVC 三层架构，各层职责边界清晰，上层依赖下层，禁止反向依赖，降低代码耦合度，便于并行开发与后期维护。
2. **配置与代码分离**：核心参数（数据库连接、事务配置、拦截规则）采用配置文件管理，避免硬编码，支持多环境切换。
3. **公共能力下沉**：统一返回、异常处理、日志记录、权限校验等通用能力通过 AOP、拦截器实现，业务代码无侵入，避免重复开发。
4. **安全与合规优先**：从架构层面兜底数据安全、访问安全、输入安全，内置 SQL 注入防护、越权访问拦截、密码加密存储等基础安全能力。
5. **可测试与可追溯**：核心业务方法支持单元测试，所有管理操作留痕，状态变更有完整历史记录，满足审计追溯要求。

## 二、整体分层架构
### 2.1 架构总览
本项目采用**经典三层单体 Web 架构**，自下而上分为数据层、持久层、业务层、表现层，同时通过 Spring AOP 与拦截器实现横切关注点的统一处理，整体架构无额外分布式组件，适配学生团队开发能力与两周交付周期。

```
┌───────────────────────────────────────────────────┐
│                   视图层 (View)                    │
│  JSP + Bootstrap / 可选 Vue 前端页面交互展示       │
└─────────────────┬─────────────────────────────────┘
                  │ HTTP 请求/响应
┌─────────────────▼─────────────────────────────────┐
│                 表现层 (Controller)                │
│  Spring MVC 控制层：请求接收、参数校验、结果封装    │
└─────────────────┬─────────────────────────────────┘
                  │ 方法调用
┌─────────────────▼─────────────────────────────────┐
│                  业务层 (Service)                  │
│  业务逻辑处理、事务控制、状态流转、多表联动        │
└─────────────────┬─────────────────────────────────┘
                  │ 数据访问
┌─────────────────▼─────────────────────────────────┐
│                 持久层 (DAO/Mapper)                │
│  MyBatis Mapper 接口 + XML：SQL 映射、CRUD 操作    │
└─────────────────┬─────────────────────────────────┘
                  │ JDBC 连接
┌─────────────────▼─────────────────────────────────┐
│                  数据层 (Database)                 │
│  MySQL 8.0：业务数据持久化存储                     │
└───────────────────────────────────────────────────┘

【横切公共组件（Spring AOP / 拦截器）】
┌───────────────────────────────────────────────────┐
│  权限拦截器、操作日志切面、全局异常处理、事务管理  │
└───────────────────────────────────────────────────┘
```

### 2.2 各层职责划分
| 层级 | 核心职责 | 技术实现 | 开发约束 |
| :--- | :--- | :--- | :--- |
| 表现层 Controller | 1. 接收前端请求，参数绑定与基础校验<br>2. 调用 Service 层处理业务<br>3. 封装统一返回结果，响应前端<br>4. 页面跳转与视图渲染 | Spring MVC `@Controller` / `@RestController`、参数绑定、ModelAndView | 禁止直接操作数据库；禁止编写复杂业务逻辑；仅做请求转发与结果封装 |
| 业务层 Service | 1. 核心业务逻辑实现，状态流转控制<br>2. 调用 DAO 层进行数据操作<br>3. 声明式事务管理，保证数据一致性<br>4. 业务参数合法性校验 | Spring 管理 Bean、`@Service`、`@Transactional` 声明式事务 | 禁止直接处理 HTTP 请求与响应；复杂业务必须加事务注解；禁止拼接 SQL |
| 持久层 DAO/Mapper | 1. 数据库 CRUD 操作封装<br>2. 动态 SQL 编写、多表关联查询<br>3. 结果集与实体类映射 | MyBatis Mapper 接口 + XML 映射文件、`#{} `预编译 | 禁止编写业务逻辑；SQL 全部使用预编译；禁止使用 `${}` 拼接非必要字段 |
| 横切组件 | 1. 登录校验、权限拦截<br>2. 操作日志自动记录<br>3. 全局异常统一捕获<br>4. 统一事务管理 | Spring AOP、HandlerInterceptor、`@ControllerAdvice` | 与业务代码解耦；仅处理通用逻辑；禁止侵入业务核心流程 |

### 2.3 请求处理全流程
1. 客户端发起 HTTP 请求，由 Tomcat 容器接收，转发至 Spring MVC 前端控制器 `DispatcherServlet`。
2. 前端控制器匹配对应 Controller 方法，先执行拦截器链：登录校验 → 权限校验，校验不通过直接返回错误或跳转登录页。
3. 拦截通过后，进入 Controller 层，完成参数绑定、基础格式校验，调用对应 Service 接口。
4. Service 层执行业务逻辑，通过声明式事务控制数据一致性，调用 DAO 层访问数据库。
5. DAO 层通过 MyBatis 执行 SQL，将数据结果返回至 Service 层，再逐层返回至 Controller。
6. Controller 封装统一结果对象，若为页面请求则经视图解析器渲染 JSP 页面；若为 AJAX 请求则直接返回 JSON 数据。
7. 请求全程由全局异常处理器捕获异常，统一封装错误信息返回；增删改操作由 AOP 切面自动记录操作日志。

## 三、工程结构设计
### 3.1 Maven 项目包结构
采用 Maven 单工程分层结构，包名遵循反向域名规范，按业务模块与技术分层双重划分，便于 3 人并行开发，职责清晰。

```
com.h.dormitory
├── common                # 公共基础组件（全员共用）
│   ├── result            # 统一返回结果封装
│   ├── exception         # 自定义异常、全局异常处理
│   ├── constant          # 常量定义（状态枚举、角色枚举）
│   ├── utils             # 工具类（MD5加密、日期处理、分页工具）
│   └── aop               # Spring AOP 切面（操作日志、性能监控）
│
├── interceptor           # 拦截器（登录拦截、权限拦截）
│
├── entity                # 数据库实体类（与表一一对应）
│
├── dao                   # MyBatis Mapper 接口
│
├── dto                   # 数据传输对象（接收前端请求参数）
│
├── vo                    # 视图对象（返回前端展示数据）
│
├── service               # 业务层接口
│   └── impl              # 业务层实现类
│
└── controller            # 控制层
    ├── admin             # 管理员端控制器
    ├── dorm              # 宿管端控制器
    └── student           # 学生端控制器
```

**资源目录结构**：
```
src/main/resources
├── spring                # Spring 核心配置
│   ├── spring-dao.xml    # 数据源、SqlSessionFactory、Mapper扫描、连接池配置
│   ├── spring-service.xml # 组件扫描、事务管理器、事务注解驱动
│   └── spring-mvc.xml    # MVC注解驱动、视图解析器、拦截器注册、静态资源放行
├── mybatis
│   └── mybatis-config.xml # MyBatis 核心配置（驼峰命名、日志、别名）
├── mapper                # MyBatis XML 映射文件，按模块分子目录
├── config
│   └── jdbc.properties   # 数据库连接配置
└── log4j.properties      # 日志配置
```

### 3.2 配置文件规划
采用「**注解 + XML 混合配置**」模式，兼顾开发效率与配置可读性，贴合课程教学内容。
1. **Bean 管理**：业务类（Controller、Service、DAO）通过注解 `@Controller`、`@Service`、`@Repository` 注册，组件扫描自动装配。
2. **核心框架配置**：数据源、事务管理器、视图解析器、拦截器等通过 XML 配置，便于统一管理与排查问题。
3. **参数配置**：数据库账号密码、连接池参数等外置到 properties 文件，支持多环境切换。
4. **MyBatis 映射**：SQL 语句全部写在 XML 文件中，与 Java 代码分离，便于 SQL 优化与维护。

## 四、核心模块架构设计
### 4.1 公共基础组件模块
**定位**：全项目通用能力底座，统一开发范式，降低重复代码。
1. **统一返回结果 `Result`**：封装状态码、提示信息、返回数据，所有接口返回格式统一，便于前端统一处理。
2. **全局异常处理器**：基于 `@ControllerAdvice` 实现，捕获业务异常、参数校验异常、系统异常，统一返回错误信息，避免堆栈信息泄露。
3. **自定义异常类**：`BusinessException` 业务异常，用于业务逻辑错误抛出，支持自定义错误码与提示信息。
4. **常量与枚举**：统一管理角色类型、床位状态、报修状态、审批状态等枚举值，避免魔法值。
5. **通用工具类**：MD5 密码加密工具、日期转换工具、分页参数工具、Session 工具。

### 4.2 用户认证与权限模块
**架构核心**：Session 身份认证 + 拦截器角色权限校验，轻量易实现，适配单体应用场景。
1. **认证流程**：
   - Controller 接收登录账号密码，调用 Service 层查询用户信息
   - Service 层校验密码（MD5 加密比对），校验账号状态
   - 校验通过后，将用户信息存入 HttpSession，返回登录成功
   - 登出时清除 Session 中的用户信息
2. **权限拦截实现**：
   - 实现 `HandlerInterceptor` 接口，定义 `LoginInterceptor` 登录拦截器：校验 Session 中是否存在用户信息，不存在则跳转登录页
   - 定义 `PermissionInterceptor` 权限拦截器：根据用户角色与请求路径匹配权限，越权访问返回 403 错误
   - 在 spring-mvc.xml 中注册拦截器，配置放行路径（登录页、静态资源、验证码等）
3. **数据隔离**：
   - 学生角色：仅能查询、操作本人相关数据（本人报修、本人晚归记录等）
   - 宿管角色：仅能操作所负责楼栋的业务数据
   - 管理员角色：全量数据操作权限

### 4.3 宿舍基础信息模块
**架构核心**：三级级联数据结构，分层 CRUD，关联校验。
1. **数据层级**：楼栋 → 房间 → 床位，三级从属关系，通过外键逻辑关联。
2. **分层实现**：
   - 持久层：单表 CRUD + 动态条件查询 + 级联查询（查询楼栋下所有房间）
   - 业务层：处理数据校验（删除楼栋时校验是否存在房间/在住学生）、批量操作（批量初始化床位）
   - 控制层：提供分页查询、新增、编辑、删除接口，参数校验后调用 Service
3. **核心约束**：删除上级数据前必须校验下级是否存在关联数据，存在则禁止删除，避免脏数据；新增房间/床位时校验编号唯一性。

### 4.4 入住与调宿业务模块
**架构核心**：事务驱动的状态流转，多表原子操作，保证数据一致性。
1. **入住办理**：
   - Service 方法添加 `@Transactional` 事务注解
   - 原子操作：校验床位空闲状态 → 新增入住记录 → 更新床位状态为「已入住」
   - 任意一步失败则事务回滚，避免数据不一致
2. **退宿办理**：
   - 原子操作：更新入住记录状态为「已退宿」、填写退宿时间 → 对应床位状态置为「空闲」
3. **调宿审批**：
   - 审批通过时原子执行：原床位释放空闲 → 目标床位置为已入住 → 更新原入住记录 → 生成新入住记录 → 更新申请单状态
   - 审批驳回：仅更新申请单状态与审批意见，不改动住宿数据
4. **数据查询**：支持多维度关联查询（学生信息 + 床位 + 房间 + 楼栋），通过 MyBatis 多表关联查询 + VO 对象封装返回。

### 4.5 日常运维业务模块
包含报修管理、晚归登记、访客登记三个子模块，架构模式统一。
1. **共性设计**：
   - 单表为主，关联用户、楼栋基础信息
   - 支持状态流转、条件筛选、分页查询
   - 增删改操作自动记录操作日志
2. **报修模块**：支持状态从「待处理 → 处理中 → 已完成」流转，处理人、处理时间自动回填，学生端仅查看本人报修，宿管端仅查看本楼栋报修。
3. **晚归/访客模块**：宿管端录入，按楼栋权限隔离，支持按时间范围、学号筛选查询，管理员全局可查。

### 4.6 操作日志审计模块
**架构核心**：Spring AOP 切面无侵入实现，业务代码零感知。
1. **实现方式**：定义 `@OperLog` 自定义注解，标注在 Controller 层增删改方法上；定义操作日志切面，通过环绕通知拦截注解标记的方法。
2. **日志记录逻辑**：
   - 拦截方法执行，获取操作人（Session 中获取）、操作模块、操作类型、请求参数
   - 方法执行成功后，记录操作描述、操作时间、操作 IP
   - 异步写入 `sys_oper_log` 数据库表，不影响主业务性能
3. **追溯能力**：管理员可在后台查看操作日志列表，支持按操作人、模块、时间筛选，满足审计要求。

## 五、关键技术点详细设计
### 5.1 Spring MVC 控制层设计
1. **请求映射**：采用 RESTful 风格路径，按模块划分请求前缀，如 `/admin/building`、`/student/repair`。
2. **参数绑定**：支持表单参数、JSON RequestBody、路径变量、RequestParam 多种方式，日期类型通过 `@DateTimeFormat` 注解统一转换。
3. **视图解析**：配置 InternalResourceViewResolver，前缀 `/WEB-INF/jsp/`，后缀 `.jsp`，页面跳转通过返回字符串逻辑视图名实现。
4. **静态资源处理**：配置 `<mvc:resources>` 放行 js、css、图片等静态资源，避免被前端控制器拦截。
5. **参数校验**：后端采用 Hibernate Validator 注解校验（可选），基础非空、长度校验在 Controller 层完成，业务合法性校验在 Service 层完成。

### 5.2 MyBatis 持久层设计
1. **开发模式**：采用 Mapper 接口 + XML 映射文件模式，接口与 XML 同包同名自动映射。
2. **SQL 规范**：
   - 所有参数使用 `#{} ` 预编译，杜绝 SQL 注入风险；仅动态排序、动态表名场景可使用 `${}`，且必须做白名单校验
   - 条件查询使用 `<if>`、`<where>` 动态 SQL 标签，避免拼接冗余条件
   - 批量操作使用 `<foreach>` 标签，提升执行效率
3. **结果映射**：简单单表查询使用自动映射，多表关联查询使用 `resultMap` 显式映射字段，支持级联对象封装。
4. **分页实现**：采用物理分页，DAO 层传入页码、每页条数参数，SQL 中使用 LIMIT 实现；可选集成 PageHelper 插件简化分页开发。
5. **主键回填**：新增操作使用 `useGeneratedKeys="true"` 回填自增主键，便于后续业务逻辑使用。

### 5.3 Spring AOP 切面设计
1. **核心切面 1：操作日志切面**
   - 切点：标注了 `@OperLog` 注解的 Controller 方法
   - 通知类型：环绕通知
   - 核心作用：自动记录管理端操作日志，业务代码无侵入
2. **核心切面 2：性能监控切面（可选）**
   - 切点：所有 Service 层方法
   - 通知类型：环绕通知
   - 核心作用：统计方法执行耗时，日志输出慢方法，便于性能优化
3. **AOP 配置**：在 spring-service.xml 中开启 `<aop:aspectj-autoproxy>` 注解驱动，切面类交由 Spring 容器管理。

### 5.4 声明式事务管理设计
1. **配置方式**：在 spring-service.xml 中配置 `DataSourceTransactionManager` 事务管理器，开启 `<tx:annotation-driven>` 注解驱动。
2. **使用方式**：在 Service 层的核心业务方法上添加 `@Transactional` 注解，默认遇到 RuntimeException 回滚。
3. **事务属性**：
   - 传播行为：默认 `REQUIRED`，有事务则加入，无则新建事务
   - 隔离级别：默认使用数据库隔离级别（MySQL 可重复读）
   - 只读属性：纯查询方法设置 `readOnly = true`，提升性能
4. **事务覆盖范围**：入住办理、退宿办理、调宿审批、批量删除等涉及多表写操作的业务必须添加事务注解，保证数据一致性。

### 5.5 统一返回与全局异常处理
1. **统一返回结构 `Result<T>`**：
   - `code`：状态码，200 成功，其他为错误码
   - `msg`：提示信息
   - `data`：返回数据
2. **全局异常处理**：
   - 捕获 `BusinessException`：返回业务错误码与提示信息
   - 捕获 `MethodArgumentNotValidException`：返回参数校验错误信息
   - 捕获 `Exception`：返回系统繁忙提示，记录错误日志，不暴露堆栈信息
3. **错误码规范**：
   - 200：操作成功
   - 400：参数错误
   - 401：未登录
   - 403：无权限
   - 500：系统内部错误
   - 1000+：业务自定义错误码

## 六、安全架构设计
### 6.1 数据安全
1. **密码加密存储**：用户密码采用 MD5 + 盐值加密存储，禁止明文保存数据库，登录时加密后比对。
2. **敏感数据脱敏**：身份证号、手机号等敏感信息，前端展示时做脱敏处理（中间字符替换为*）。
3. **数据备份**：数据库支持定期备份，核心业务表禁止物理删除，采用逻辑删除或保留历史记录。

### 6.2 访问安全
1. **登录认证**：所有业务接口必须经过登录拦截器校验，未登录用户无法访问业务资源。
2. **权限隔离**：基于角色的接口级权限校验，学生无法访问管理员、宿管接口；宿管无法操作非负责楼栋数据。
3. **越权访问防护**：学生查询个人数据时，强制从 Session 中获取用户 ID，不接收前端传入的学生 ID 参数，杜绝水平越权。

### 6.3 输入安全
1. **SQL 注入防护**：MyBatis 全部使用 `#{} ` 预编译语句，禁止直接拼接 SQL 字符串。
2. **XSS 防护**：后端对用户输入的文本内容做特殊字符过滤，前端页面输出时做转义处理。
3. **参数校验**：所有入参做长度、格式、范围校验，非法参数直接拦截，避免异常数据入库。

## 七、部署与运行架构
### 7.1 部署拓扑

## 八、工程规范与质量保障
### 8.1 代码开发规范
1. 严格遵循《阿里巴巴 Java 开发手册》，命名见名知意，禁止拼音、无意义变量名。
2. 类、方法、核心逻辑必须添加注释，复杂业务说明实现思路与注意事项。
3. 分层调用严格遵循：Controller → Service → DAO，禁止跨层调用、反向调用。
4. 禁止在业务代码中硬编码配置参数，统一通过常量或配置文件管理。

### 8.2 接口设计规范
1. 所有接口统一返回 `Result` 封装对象，禁止直接返回字符串、Map 等不规范格式。
2. 分页查询统一接收 `pageNum`、`pageSize` 参数，返回分页数据与总条数。
3. 新增、编辑接口参数校验完整，返回明确的错误提示信息。
4. 接口路径按角色+模块划分，语义清晰，如 `/admin/room/list`、`/student/repair/add`。

### 8.3 质量管控机制
1. **代码审核**：所有功能合并至开发分支前，由组长进行代码审核，重点检查 SQL 规范、权限校验、事务处理、代码分层。
2. **单元测试**：核心 Service 方法编写单元测试，验证业务逻辑正确性与事务有效性。
3. **集成测试**：联调阶段执行全流程测试，覆盖正常场景、边界场景、异常场景。
4. **安全测试**：专项验证越权访问、SQL 注入、参数异常等安全场景，确保基础安全能力生效。

## 九、性能与容错设计
### 9.1 性能优化要点
1. **数据库优化**：核心查询字段建立索引，避免全表扫描；优化慢查询 SQL，减少不必要的多表关联。
2. **连接池优化**：使用 Druid 连接池，合理配置初始连接数、最大连接数、超时时间，避免连接泄露。
3. **分页优化**：所有列表查询强制分页，避免一次性加载大量数据，减少内存占用与传输耗时。
4. **静态资源**：前端静态资源压缩，合理利用浏览器缓存，提升页面加载速度。

### 9.2 异常容错机制
1. **全局异常捕获**：所有异常统一处理，系统异常返回友好提示，记录详细错误日志，避免页面白屏。
2. **参数兜底**：入参设置默认值，空数据场景做兼容处理，避免空指针异常。
3. **事务回滚**：核心写操作全部纳入事务管理，异常时自动回滚，保证数据一致性。
4. **优雅降级**：非核心功能（如操作日志写入）异常时，不影响主业务流程正常执行。

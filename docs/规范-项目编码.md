# 高校公寓管理系统 Java 编码规范
## 一、代码风格规范
### 1.1 命名规范
- 类名：大驼峰命名法，见名知意，禁止拼音、无意义命名
- 方法名、变量名：小驼峰命名法，动词开头表述动作
- 常量：全大写+下划线分隔，使用 `static final` 修饰
- 包名：全小写，按业务模块分层，禁止大写与下划线
- 集合变量：明确表述元素类型，如 `studentList` 而非 `list`

### 1.2 代码格式
- 统一使用 4 空格缩进，禁止 Tab
- 方法长度原则上不超过 80 行，超长方法必须拆分
- 所有集合指定泛型，禁止使用无泛型的原始类型
- 左大括号不换行，右大括号单独一行，遵循标准 Java 代码格式

### 1.3 注释规范
- 类、公共方法必须添加 Javadoc 注释，说明功能、入参、返回值、抛出异常
- 核心业务逻辑、复杂算法必须添加行内注释，说明实现思路与注意事项
- 禁止无意义注释，禁止注释掉的代码块提交至仓库
- 常量、枚举值必须添加注释说明含义

### 1.4 文件结构
- 严格遵循项目既定 MVC 分层包结构，禁止随意新增跨层包
- 同类文件按模块归集，禁止零散放置公共工具类

## 二、异常处理规范
### 2.1 捕获原则
- 必须捕获具体异常类型，禁止使用 `catch (Exception e)` 兜底捕获所有异常，全局异常处理器除外
- 禁止空 catch 块，至少输出错误日志，禁止吞掉异常
- 所有可能抛出异常的操作（数据库操作、文件IO、类型转换、外部调用）必须做异常处理

### 2.2 分层处理规则
- 当前层可处理的异常，必须本地处理（重试、降级、返回默认值、转化为业务错误），严禁向上层抛出未处理的原生异常
- 无法本地处理的业务异常，统一封装为自定义 `BusinessException` 后抛出，必须携带错误码与提示信息
- 系统级异常禁止直接抛给前端，由全局异常处理器统一捕获、封装、返回

### 2.3 事务与数据一致性
- 所有多表写操作、核心业务流程必须添加 `@Transactional` 声明式事务
- 不可逆操作（数据删除、状态变更、批量更新）执行前必须做防御性校验，确保失败无脏数据残留
- 业务异常必须触发事务回滚，禁止异常被吞导致数据不一致

### 2.4 异常输出约束
- 禁止将异常堆栈、系统底层错误信息直接返回给前端用户
- 用户侧仅返回业务错误码与友好提示，完整堆栈仅记录到日志
- 禁止在前端输出数据库表名、SQL 片段、配置参数等敏感信息

### 2.5 外部依赖约束
- 数据库连接、文件操作、外部接口调用必须设置超时时间
- 外部依赖异常必须有降级处理逻辑，禁止因外部依赖失败导致系统不可用

### 2.6 空值与边界校验
- 所有公共方法入参必须做空值校验，禁止直接使用可能为 null 的对象
- 分页参数、下标、数值范围必须做边界合法性校验，非法参数直接拦截

## 三、日志规范
### 3.1 日志框架
- 统一使用 SLF4J 日志门面 + Logback 实现，禁止使用 `System.out.println()` 输出内容
- 每个类通过 `LoggerFactory.getLogger(当前类.class)` 获取日志对象

### 3.2 日志分级
- DEBUG：开发调试信息，仅开发环境启用，生产环境关闭
- INFO：业务关键节点记录（登录成功、入住办理、审批完成、数据新增/修改），必须包含操作人、业务ID
- WARN：非预期但可恢复的异常、参数不合法、边界场景触发，不阻断主流程
- ERROR：系统异常、业务失败、事务回滚、外部依赖不可用，必须输出完整异常堆栈

### 3.3 日志记录规则
- 异常日志必须使用 `logger.error("错误描述", e)` 格式，输出完整堆栈，禁止仅输出异常消息
- 业务日志必须包含关键标识（用户ID、业务单号、模块名），便于问题追溯
- 禁止输出敏感信息：用户密码、身份证号、手机号明文，必须脱敏后输出
- 禁止在循环内频繁打印日志，避免性能损耗

## 四、硬编码治理规范
### 4.1 治理原则
- 常量与使用方在同模块定义，禁止创建跨模块的全局“上帝”常量类
- 所有可配置项必须提供代码默认值，避免配置缺失导致启动失败
- 业务状态、类型集合优先使用 `Enum` 枚举定义，禁止魔法数字、魔法字符串

### 4.2 分类处理规则
| 类型 | 示例 | 处理方式 |
| :--- | :--- | :--- |
| 业务常量 | 角色类型、床位状态、审批状态、报修类型 | 对应模块内定义 `Enum` 枚举，或模块级 `static final` 常量 |
| 技术策略 | 默认分页大小、超时时间、重试次数 | 配置文件管理，配置类封装，代码内置默认值 |
| 环境依赖 | 数据库地址、账号、连接池参数 | properties 配置文件注入，代码中禁止硬编码具体值 |
| 临时调试 | 测试开关、临时输出语句 | 提交代码前必须删除，禁止遗留 |

### 4.3 禁止项
- 禁止业务状态、类型直接使用数字/字符串字面量，如 `if(status == 1)` 必须替换为枚举
- 禁止散落硬编码 URL、硬编码配置值在业务代码中
- 禁止创建单一全局常量类归集所有模块无关常量

## 五、分层开发规范
### 5.1 Controller 层
- 职责：接收请求、参数基础校验、调用 Service、封装统一返回结果
- 禁止：编写业务逻辑、直接操作数据库、直接输出异常堆栈
- 入参：基础非空、长度校验在 Controller 层完成
- 返回：统一返回 `Result` 封装对象，禁止直接返回实体、Map、字符串

### 5.2 Service 层
- 职责：业务逻辑实现、业务参数校验、事务控制、调用 DAO 层
- 禁止：直接操作 HttpServletRequest/Response、拼接 SQL、跨层调用 Controller
- 核心写方法必须添加事务注解，业务异常必须明确抛出
- 业务逻辑复杂时必须拆分私有方法，保证主流程清晰

### 5.3 DAO / Mapper 层
- 职责：数据库 CRUD、SQL 映射，仅做数据访问
- 禁止：编写业务逻辑、处理业务异常
- MyBatis SQL 全部使用 `#{} ` 预编译，杜绝 SQL 注入；仅动态表名、排序字段可使用 `${}`，且必须做白名单校验
- 多表关联查询使用 `resultMap` 显式映射，禁止使用 * 查询所有字段

### 5.4 调用约束
- 仅允许上层调用下层：Controller → Service → DAO
- 禁止反向依赖、跨层调用
- 同层模块间调用通过接口依赖，禁止直接依赖实现类

## 六、安全编码规范
- SQL 注入防护：MyBatis 全部使用预编译语句，禁止直接拼接 SQL 字符串
- 越权防护：学生端个人数据强制从 Session 获取用户 ID，禁止接收前端传入的学生 ID；宿管端数据按负责楼栋隔离
- 敏感数据：用户密码加密存储，禁止明文；接口返回、日志输出中身份证、手机号必须脱敏
- 输入校验：所有入参做合法性校验，过滤特殊字符，防范 XSS 风险
- 权限校验：所有业务接口必须经过权限拦截器校验，禁止绕过权限体系

## 七、代码验收检查清单
出现以下任意一项，代码审查不通过：
1. 存在空 catch 块，或捕获异常后未做任何处理
2. 直接将异常堆栈、系统错误信息返回前端
3. 核心写操作未添加事务注解，存在数据不一致风险
4. 外部依赖调用未设置超时，无异常降级处理
5. 使用魔法数字、魔法字符串表示业务状态与类型
6. MyBatis 中无校验直接使用 `${}` 拼接 SQL，存在注入风险
7. 存在越权漏洞，学生可通过参数篡改访问他人数据
8. 密码、身份证等敏感信息明文存储或明文输出日志
9. 代码中遗留 `System.out.println`、调试代码、注释掉的代码块
10. 分层混乱，跨层调用、职责越界
